---
name: Push Back on New Features
description: User wants discussion and critical thinking before implementing new feature requests — don't just build immediately
type: feedback
---

When the user drops a new feature request, do NOT jump straight into implementation.

Instead: push back, brainstorm, and discuss first. Cover:
- **Purpose** — what problem does this solve? Is it the right problem?
- **Practicality** — is this feasible given current constraints (hardware, SDK, scope)?
- **Alternatives** — are there better or simpler ways to achieve the same goal?

Only proceed to implementation after alignment.

**Why:** User found that I was implementing features too eagerly without questioning whether they were the right thing to build. This led to wasted effort (e.g., receipt capture feature added without discussing whether glasses camera is the right capture method vs phone camera).

**How to apply:** Treat every new feature request as a design discussion first. Ask clarifying questions, surface trade-offs, suggest alternatives. Get a "go ahead" before writing code.
